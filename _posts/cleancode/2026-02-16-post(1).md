AI에게 테스트 코드를 요청할 때 꼭 지켜야 할 5가지
1. 커버리지 범위를 명시할 것
AI는 기본적으로 "잘 되는 경우"만 먼저 만든다.
안 좋은 요청:

로그인 API 테스트 코드 만들어줘

좋은 요청:

로그인 API 테스트 코드 만들어줘. 다음 시나리오를 모두 포함해:

정상 로그인 (happy path)
빈 아이디/비밀번호 (입력 검증)
존재하지 않는 계정
비밀번호 불일치
5회 실패 후 계정 잠김
이미 로그인된 상태에서 재로그인


이렇게 범위를 직접 써줘야 제대로 된 테스트가 나온다.

2. 테스트 격리를 요구할 것
DB나 시간, 외부 API에 의존하면 환경에 따라 테스트 결과가 달라진다.
문제가 되는 코드:
java@Test
void 주문_생성_테스트() {
    Order order = orderService.create(userId, productId);
    // 실제 DB에 저장되고, 실제 결제 API 호출됨
    assertThat(order.getStatus()).isEqualTo("PAID");
}
요청 예시:

주문 생성 API 테스트 코드를 만들어줘.
단, DB는 전부 in-memory로, 외부 API는 mock으로 처리해줘.
시간도 고정값 사용하고.

개선된 코드:
java@Test
void 주문_생성_테스트() {
    // given - 의존성 전부 mock
    when(userRepository.findById(userId)).thenReturn(mockUser);
    when(paymentClient.charge()).thenReturn(success);
    when(clock.now()).thenReturn(fixedTime);
    
    // when
    Order order = orderService.create(userId, productId);
    
    // then
    assertThat(order.getStatus()).isEqualTo("PAID");
}

3. 테스트 이름이 문서가 되게 할 것
test1(), testSuccess() 같은 이름은 실패해도 뭐가 깨진 건지 모른다.
안 좋은 이름:
java@Test
void test1() { ... }

@Test
void loginTest() { ... }

@Test
void testFail() { ... }
좋은 이름:
java@Test
void 만료된_토큰으로_요청시_401을_반환한다() { ... }

@Test
void 존재하지_않는_상품_주문시_ProductNotFoundException을_던진다() { ... }

@Test
void 재고가_0일때_주문_생성시_OutOfStockException을_던진다() { ... }
테스트가 실패하면 이름만 봐도 어느 시나리오가 깨졌는지 바로 안다.
AI 요청 예시:

테스트 이름은 "언제_무엇을_하면_어떻게_된다" 형식으로 한글로 작성해줘.


4. 구현이 아니라 행동을 검증하게 할 것
내부 메서드 호출 횟수를 검증하면 리팩토링할 때 테스트가 깨진다.
나쁜 예 (구현 검증):
java@Test
void 주문_생성시_재고_감소() {
    orderService.create(productId, quantity);
    
    // 내부 구현에 의존
    verify(inventoryService, times(1)).decrease(productId, quantity);
    verify(eventPublisher, times(1)).publish(any());
}
→ inventoryService를 다른 방식으로 바꾸면 테스트가 깨짐
좋은 예 (행동 검증):
java@Test
void 주문_생성시_재고_감소() {
    // given
    Inventory before = inventoryRepository.findById(productId);
    
    // when
    orderService.create(productId, quantity);
    
    // then
    Inventory after = inventoryRepository.findById(productId);
    assertThat(after.getStock()).isEqualTo(before.getStock() - quantity);
}
→ 내부 로직이 바뀌어도 테스트는 여전히 유효
AI 요청 예시:

테스트는 내부 메서드 호출 횟수가 아니라, 최종 상태나 반환값을 검증하도록 작성해줘.


5. CI에서 바로 돌아가게 할 것
로컬에서만 되는 테스트는 의미가 없다.
문제가 되는 코드:
java@Test
void 파일_업로드_테스트() {
    // 내 컴퓨터 경로에 의존
    File file = new File("C:/Users/me/test.jpg");
    uploadService.upload(file);
}

@Test
void 데이터_조회_테스트() {
    // 로컬 DB에 수동으로 넣어둔 데이터 의존
    List<User> users = userRepository.findAll();
    assertThat(users).hasSize(10);
}
개선된 코드:
java@Test
void 파일_업로드_테스트() {
    // 테스트용 리소스 사용
    File file = new ClassPathResource("test-files/sample.jpg").getFile();
    uploadService.upload(file);
}

@Test
void 데이터_조회_테스트() {
    // 테스트에서 직접 데이터 준비
    userRepository.save(new User("test1"));
    userRepository.save(new User("test2"));
    
    List<User> users = userRepository.findAll();
    assertThat(users).hasSize(2);
}
AI 요청 예시:

테스트는 외부 파일이나 DB에 미리 데이터가 있다고 가정하지 말고,
테스트 코드 내에서 필요한 데이터를 전부 준비하도록 작성해줘.

마무리
AI한테 테스트 코드를 요청할 때 가장 중요한 건 구체적으로 요구하는 것이다.
"테스트 코드 만들어줘"가 아니라, 어떤 시나리오를 커버해야 하는지, 어떻게 검증해야 하는지, 어떤 의존성을 어떻게 처리해야 하는지를 명확하게 전달해야 한다.
위 5가지만 지켜도 AI가 만든 테스트 코드를 그대로 쓸 수 있는 확률이 확 올라간다.
