# AI를 활용한 테스트 코드 작성 시리즈 — 3편

## AI로 테스트 코드 PR 리뷰하기

> 테스트 코드는 작성보다 리뷰가 어렵습니다.
> "통과하니까 괜찮겠지"라는 착각을 AI와 함께 깨뜨리는 방법을 다룹니다.

---

### 들어가며

PR에 테스트 코드가 포함되어 있을 때, 리뷰어는 두 가지를 동시에 판단해야 합니다. "이 테스트가 올바른 것을 검증하는가?"와 "이 테스트가 충분한가?"입니다. 프로덕션 코드 리뷰에는 익숙하지만, 테스트 코드 리뷰에는 체계적인 기준이 없는 경우가 많습니다.

AI를 테스트 PR 리뷰에 활용하면, 놓치기 쉬운 누락 시나리오를 발견하고, 테스트 품질을 일관된 기준으로 평가할 수 있습니다. 이 글에서는 AI를 리뷰 도우미로 활용하는 실전 방법을 다룹니다.

---

### 1. 테스트 코드 리뷰가 어려운 이유

#### 1.1 프로덕션 코드 리뷰 vs 테스트 코드 리뷰

프로덕션 코드를 리뷰할 때는 "이 코드가 올바르게 동작하는가?"를 봅니다. 테스트 코드를 리뷰할 때는 한 단계 더 들어가야 합니다.

- 이 테스트가 **정말로 실패할 수 있는 상황이 존재하는가?** (항상 통과하는 테스트는 의미 없음)
- 이 테스트가 **프로덕션 코드의 핵심 동작을 검증하는가?** (구현 세부사항이 아닌)
- 빠진 시나리오는 없는가? (경계값, 에러 경로, 동시성)
- 테스트 자체에 버그는 없는가?

#### 1.2 리뷰어의 인지 부하

하나의 PR에 프로덕션 코드 200줄 + 테스트 코드 400줄이 들어오면, 테스트 코드가 더 길지만 리뷰 시간은 프로덕션 코드에 집중되기 쉽습니다. AI에게 테스트 부분의 초기 분석을 맡기면, 리뷰어는 AI가 발견한 이슈에 집중할 수 있습니다.

---

### 2. AI에게 테스트 PR 리뷰 요청하기

#### 2.1 기본 리뷰 프롬프트

```
역할: 시니어 소프트웨어 엔지니어로서 테스트 코드를 리뷰해줘.

리뷰 기준:
1. 테스트 커버리지 충분성: 정상/경계값/에러 케이스가 모두 있는가?
2. 테스트 품질: AAA 패턴, 하나의 테스트에 하나의 검증 주제
3. 테스트 신뢰성: flaky 가능성, 비결정적 요소 존재 여부
4. 가독성: 테스트 이름, 주석, 구조
5. 누락 시나리오: 테스트하지 않는 중요한 경로

출력 형식:
- 심각도 표시: 🔴 반드시 수정 / 🟡 권장 / 🟢 참고
- 각 이슈에 구체적인 코드 위치와 개선 제안 포함
- 마지막에 놓친 시나리오 목록

프로덕션 코드:
[코드]

테스트 코드:
[코드]
```

#### 2.2 프로덕션 코드와 테스트를 함께 제공하기

AI에게 테스트만 보여주면 안 됩니다. 프로덕션 코드를 함께 제공해야 "무엇을 테스트해야 하는지"를 AI가 판단할 수 있습니다.

```
프롬프트:
아래 프로덕션 코드와 테스트 코드를 함께 리뷰해줘.

프로덕션 코드의 모든 분기(branch)를 식별하고,
각 분기가 테스트에 의해 커버되는지 매핑해줘.
커버되지 않는 분기가 있다면 명시해줘.

[프로덕션 코드]

---

[테스트 코드]
```

#### 2.3 분기 커버리지 매핑 요청

```
프롬프트:
아래 Reconcile 함수의 모든 실행 경로를 분석하고,
각 경로가 테스트에 의해 검증되는지 표로 정리해줘.

| 실행 경로                    | 테스트 존재 여부 | 테스트 함수명       |
|----------------------------|---------------|-------------------|
| CR NotFound → return nil   | ✅/❌          | 함수명 또는 "없음"  |
| validation 실패 → Status 갱신| ✅/❌          | ...               |
| ...                        | ...           | ...               |
```

이 매핑 테이블은 리뷰 코멘트의 근거로 매우 유용합니다.

### 3. 리뷰에서 확인해야 할 7가지 패턴

AI에게 다음 안티패턴을 탐지하도록 요청할 수 있습니다.

#### 3.1 항상 통과하는 테스트 (Evergreen Test)

```go
// 🔴 이 테스트는 절대 실패하지 않는다
func TestSomething(t *testing.T) {
    result, _ := DoSomething()  // 에러를 무시
    assert.NotNil(t, result)    // nil 아니면 무조건 통과
}
```

**AI 리뷰 포인트**: "에러 반환값을 무시하고 있으며, assertion이 `NotNil`로만 되어 있어 의미 있는 검증이 아닙니다. 구체적인 값 비교가 필요합니다."

#### 3.2 구현에 결합된 테스트 (Implementation-Coupled Test)

```go
// 🟡 내부 구현이 바뀌면 테스트도 깨진다
func TestReconcile(t *testing.T) {
    // Mock에서 정확히 3번 호출되었는지 검증
    mockClient.AssertNumberOfCalls(t, "Get", 3)
    mockClient.AssertNumberOfCalls(t, "Update", 1)
}
```

**AI 리뷰 포인트**: "호출 횟수는 구현 세부사항입니다. 최종 상태(리소스가 생성되었는지, Status가 업데이트되었는지)를 검증하는 것이 더 견고합니다."

#### 3.3 비결정적 테스트 (Flaky Test)

```go
// 🔴 타이밍에 따라 실패할 수 있다
func TestEventualConsistency(t *testing.T) {
    go startReconcile()
    time.Sleep(2 * time.Second)  // 2초면 충분할까?
    assert.Equal(t, "Ready", getStatus())
}
```

**AI 리뷰 포인트**: "`time.Sleep`은 비결정적입니다. `Eventually`(Gomega) 또는 polling 패턴으로 교체하세요."

#### 3.4 과도한 Mock (Over-Mocking)

```go
// 🟡 실제로 무엇을 테스트하는 건지 불분명
func TestHandler(t *testing.T) {
    mockDB := new(MockDB)
    mockCache := new(MockCache)
    mockLogger := new(MockLogger)
    mockMetrics := new(MockMetrics)
    mockAuth := new(MockAuth)
    // 5개의 Mock 설정 코드 50줄...
    // 실제 테스트 코드 3줄
}
```

**AI 리뷰 포인트**: "Mock이 5개 이상이면 테스트 대상의 책임이 과도하거나, integration 테스트가 더 적합할 수 있습니다."

#### 3.5 테스트 간 숨겨진 의존성

```go
// 🔴 TestA가 먼저 실행되어야 TestB가 통과
var sharedState string

func TestA(t *testing.T) {
    sharedState = "initialized"
}

func TestB(t *testing.T) {
    assert.Equal(t, "initialized", sharedState)
}
```

#### 3.6 불필요하게 넓은 Scope

```go
// 🟡 수십 개의 assertion이 하나의 테스트에
func TestEntireWorkflow(t *testing.T) {
    // CR 생성 검증
    // StatefulSet 생성 검증
    // Service 생성 검증
    // Ingress 생성 검증
    // Status 업데이트 검증
    // 스케일 업 검증
    // 스케일 다운 검증
    // CR 삭제 검증
}
```

**AI 리뷰 포인트**: "하나의 테스트가 8가지 동작을 검증합니다. 중간에 실패하면 이후 검증은 건너뛰어, 어디서 문제가 생겼는지 파악이 어렵습니다."

#### 3.7 에러 메시지 없는 Assertion

```go
// 🟡 실패 시 "expected true, got false"만 출력
assert.True(t, isValid)

// ✅ 실패 시 맥락을 알 수 있다
assert.True(t, isValid, "CR with valid spec should pass validation")
```

### 4. AI 리뷰 워크플로우

#### 4.1 PR 리뷰 시 AI 활용 단계

```
Step 1: 변경된 프로덕션 코드의 분기 분석 요청
        → AI가 모든 실행 경로를 나열

Step 2: 테스트 코드와 분기 매핑 요청
        → AI가 커버되지 않는 경로를 식별

Step 3: 안티패턴 탐지 요청
        → AI가 위 7가지 패턴을 스캔

Step 4: 누락 시나리오에 대한 테스트 초안 요청
        → AI가 추가 테스트 코드 제안

Step 5: 사람이 최종 판단
        → AI 분석 결과를 바탕으로 리뷰 코멘트 작성
```

#### 4.2 실전 프롬프트 — PR diff 리뷰

```
프롬프트:
아래는 PR의 diff입니다.
프로덕션 코드 변경 부분과 테스트 코드 변경 부분이 모두 포함되어 있습니다.

다음을 분석해줘:

1. 프로덕션 코드에 새로 추가된 분기 중
   테스트에 의해 커버되지 않는 것
2. 기존 테스트 중 프로덕션 코드 변경으로 인해
   더 이상 유효하지 않을 수 있는 것
3. 테스트 코드의 안티패턴
   (flaky, over-mocking, evergreen, implementation-coupled)

리뷰 코멘트 형식으로 출력해줘:
- 파일명:라인 — [심각도] 내용
- 제안: 구체적인 개선 방안

[PR diff 붙여넣기]
```

#### 4.3 리뷰 코멘트 작성 도우미

AI가 발견한 이슈를 PR 코멘트로 다듬을 때:

```
프롬프트:
아래 테스트 이슈를 PR 리뷰 코멘트로 다듬어줘.
톤: 건설적이고 구체적. 비난이 아닌 제안 형태.
반드시 "왜 문제인지"와 "어떻게 개선할지" 포함.

이슈:
- TestReconcile_Success에서 err를 체크하지 않음
- Status 업데이트 실패 경로에 대한 테스트 없음
- time.Sleep(5s) 사용으로 flaky 가능성
```

### 5. AI 리뷰의 한계와 사람의 역할

AI는 구조적, 패턴 기반 분석에 강합니다. 하지만 다음은 사람이 판단해야 합니다.

| AI가 잘하는 것 | 사람이 해야 하는 것 |
|--------------|------------------|
| 분기 커버리지 매핑 | 비즈니스 로직의 중요도 판단 |
| 안티패턴 탐지 | "이 시나리오가 실제로 발생하는가?" |
| 누락 시나리오 제안 | 팀 컨벤션과의 일관성 |
| 코드 구조 분석 | 테스트 전략의 방향성 |
| assertion 강도 검사 | 리팩토링 시 테스트 변경 범위 판단 |

AI가 "이 분기가 테스트되지 않았습니다"라고 알려줘도, 그 분기가 테스트할 가치가 있는지는 사람이 결정합니다. 예를 들어, 발생 확률이 극히 낮은 에러 경로보다 핵심 비즈니스 로직의 경계값이 더 중요할 수 있습니다.

### 6. 팀에 AI 리뷰 문화 도입하기

#### 6.1 PR 템플릿에 AI 리뷰 체크리스트 추가

```markdown
## 테스트 리뷰 체크리스트 (AI 보조)

- [ ] AI 분기 매핑 완료: 모든 새로운 분기가 테스트에 의해 커버됨
- [ ] AI 안티패턴 스캔 완료: flaky, evergreen, over-mocking 없음
- [ ] AI 누락 시나리오 확인: 추가 필요한 시나리오 없음 또는 사유 기록
- [ ] 사람 최종 확인: AI 제안 중 반영할 것과 무시할 것 결정
```

#### 6.2 점진적 도입

- **1단계**: 개인이 PR 올리기 전 AI로 자가 리뷰
- **2단계**: 리뷰어가 AI 분석 결과를 참고하여 리뷰
- **3단계**: CI에 AI 리뷰 자동화 (→ 4편에서 다룸)

---

### 마무리

테스트 코드 PR 리뷰는 "통과하니까 OK"가 아닙니다. AI는 리뷰어의 눈이 되어 누락된 시나리오를 찾고, 안티패턴을 탐지하고, 분기 커버리지를 매핑해줍니다. 하지만 최종 판단은 사람의 몫입니다.

핵심 정리:
- 프로덕션 코드와 테스트 코드를 함께 AI에게 제공해야 의미 있는 리뷰가 나온다
- 분기 커버리지 매핑은 AI가 특히 잘하는 영역이다
- 7가지 안티패턴을 기준으로 자동 탐지를 요청하라
- AI의 리뷰 결과는 사람이 비즈니스 맥락에서 최종 판단해야 한다

> 다음 4편에서는 이 모든 것을 CI/CD 파이프라인에 자동화하는 방법을 다룹니다.
