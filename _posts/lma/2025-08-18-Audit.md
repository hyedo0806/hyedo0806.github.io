# Spring Data JPA의 Auditing 
Spring Data는 엔티티의 생성 및 수정 시점과 이를 수행한 사용자를 자동으로 추적하는 기능을 제공합니다. 이를 통해 데이터 변경 이력을 투명하게 관리할 수 있습니다.<br>
**주요 기능**
- @CreatedBy: 엔티티를 생성한 사용자
- @LastModifiedBy: 엔티티를 마지막으로 수정한 사용자
- @CreatedDate: 엔티티가 생성된 시점
- @LastModifiedDate: 엔티티가 마지막으로 수정된 시점

# Auditing 활성화 방법
### 어노테이션 기반 설정
@EnableJpaAuditing 어노테이션을 사용하여 JPA Auditing을 활성화합니다.
```
@Configuration
@EnableJpaAuditing
public class JpaConfig {
    // 필요한 빈 설정
}
```

# AuditorAware 인터페이스 구현
@CreatedBy와 @LastModifiedBy 어노테이션을 사용할 경우, 현재 사용자를 추적하기 위해 AuditorAware 인터페이스를 구현해야 합니다.
```
@Component
public class AuditorAwareImpl implements AuditorAware<String> {
    @Override
    public Optional<String> getCurrentAuditor() {
        // 현재 로그인한 사용자 반환
        return Optional.of("admin");
    }
}
```

# 날짜 및 시간 설정
엔티티의 생성 및 수정 시점을 자동으로 설정하려면 @CreatedDate와 @LastModifiedDate 어노테이션을 사용합니다. 이때, 날짜 및 시간 값을 제공하는 DateTimeProvider를 설정할 수 있습니다.
```
@Configuration
@EnableJpaAuditing(dateTimeProviderRef = "dateTimeProvider")
public class JpaConfig {
    @Bean
    public DateTimeProvider dateTimeProvider() {
        return () -> Optional.of(TemporalAccessor.from(Instant.now()));
    }
}
```
# 엔티티에 Auditing 메타데이터 추가
### 직접 추가 
```
@Entity
public class Customer {
    @CreatedBy
    private String createdBy;

    @CreatedDate
    private Instant createdDate;
}
```
### 임베디드 객체 사용
```
@Entity
public class Customer {
    private AuditMetadata auditMetadata;
}

@Embeddable
public class AuditMetadata {
    @CreatedBy
    private String createdBy;

    @CreatedDate
    private Instant createdDate;
}
```

# Auditing과 일반 log service의 차이점
## 1. Spring Data JPA Auditing
### 특징
- DB 중심: 엔티티 단위의 생성/수정/삭제 시점을 자동 기록
- 자동 관리: @CreatedDate, @LastModifiedDate, @CreatedBy, @LastModifiedBy로 자동 기록
- 조회 로그 아님: 기본적으로 엔티티 변경 시점에만 기록
- 데이터 추적: 누가 언제 무엇을 변경했는지 DB에서 직접 조회 가능<br>
✅ 장점: DB 조회 가능, 데이터 변경 추적 완전 자동
❌ 단점: 조회만 하는 경우나 실시간 로그 분석에는 부적합

### 사용 예
- SecretAccessLog처럼 영구적으로 저장해야 하는 이벤트
- IP 접근 기록, Secret 조회 시도 기록 등
- 추후 감사(audit)나 컴플라이언스 대응 필요 시
```
@Entity
@EntityListeners(AuditingEntityListener.class)
public class SecretAccessLog {
    @CreatedDate
    private LocalDateTime createdAt;

    @CreatedBy
    private String createdBy;
}
```

## 2. 일반 로그 서비스 (Logback, Log4j2 등)
### 특징
- 파일/콘솔/외부 시스템 중심: stdout, 파일, ELK, OpenTelemetry 등
- 실시간 기록: 이벤트 발생 즉시 기록 가능
- 다양한 레벨 지원: ERROR, WARN, INFO, DEBUG 등
- 조회 용이: Kibana, Grafana 등 시각화 툴과 연계 가능
- 영속성 없음: 기본적으로 DB 저장 아님, 필요하면 별도 처리<br>
✅ 장점: 실시간 모니터링, 다양한 출력 형식, 로그 레벨 관리
❌ 단점: DB에서 직접 감사(audit) 조회는 어려움, 장기 보관 필요시 별도 수단 필요
### 사용 예
- Controller 진입 로그, API 호출 응답 시간 기록
- 필터 단계에서의 IP 허용/거부 로그
- 예외 발생 시 스택 트레이스 기록
```
@Slf4j
@Service
public class SecretAccessService {
    public String fetchSecret(String accountId, String secretPath) {
        log.info("Fetching secret for account={} secret={}", accountId, secretPath);
    }
}
```

| 구분     | JPA Auditing              | 일반 로그(Logback/Log4j/OTel)           |
| ------ | ------------------------- | ----------------------------------- |
| 목적     | 데이터 변경 추적, 감사(audit)      | 실시간 이벤트/상태 모니터링                     |
| 저장 위치  | DB                        | 콘솔, 파일, ELK/OpenTelemetry           |
| 자동화 수준 | 엔티티 변경 시 자동               | 개발자가 직접 로그 코드 작성                    |
| 적합 사례  | SecretAccessLog, IP 접근 기록 | Controller 호출, 필터 단계, 예외, 트랜잭션 시간 등 |
| 단점     | 조회만은 어렵고 DB 부담            | 장기 감사용 저장에는 추가 DB 필요                |
