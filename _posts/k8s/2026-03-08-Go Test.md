# Go Test 방법
1. ``testing.T``+testify 방식 (Ginkgo 미사용)

- func TestXxx(t *testing.T) 형태
- 함수 하나가 테스트 하나이며 직관적이고, 외부 인프라없이 바로 실행
```
func TestValidate(t *testing.T) {
    cr := &brokerv1beta1.ActiveMQArtemis{...}
    valid, retry := ri.validate(cr, k8sClient, *namer)
    assert.False(t, valid)
    assert.False(t, retry)
}
```

2. Ginkgo + Gomega 방식

- envtest 기반 통합 테스트
- ``Describe --> Context --> It`` 으로 계층 구조 
- ``BeforeEach/AfterEach`` 으로 공통 setup 공유
- ``Eventually`` 로 비동기 상태 변화 폴링
```s
var _ = Describe("artemis controller", func() {
    BeforeEach(func() { ... })
    Context("tls secret reuse", func() {
        It("console and acceptor share one secret", func() {
            Expect(k8sClient.Create(ctx, cr)).Should(Succeed())
            Eventually(func(g Gomega) {
                g.Expect(k8sClient.Get(ctx, key, &obj)).To(Succeed())
            }, timeout, interval).Should(Succeed())
        })
    })
})
```

# 어떤 경우에 어떤 방식을 사용하게 되는가?
핵심 기준은 ``K8s API 서버와의 상호작용``이 필요한가

### testing.T를 쓰는 경우
**순수 로직 테스트**

validate() 함수는 CR 객체의 필드값만 검사하는 순수 함수입니다. K8s에 뭔가를 만들거나 기다릴 필요가 없어요. DuplicateKeyIn()도 문자열 파싱 로직이고, hexShaHashOfMap()도 해시 계산일 뿐입니다. 이런 함수는 입력 → 출력이 즉시 결정되므로 testing.T로 충분합니다.

### Ginkgo를 쓰는 경우
**K8s Reconcile 루프를 거치는 통합 테스트**

 "CR을 생성하면 → 컨트롤러가 Reconcile하여 → StatefulSet이 만들어지고 → Service가 생기고 → Route가 생긴다"를 검증해야 합니다. 이 과정은 비동기입니다. CR을 Create한 직후에 StatefulSet이 바로 존재하지 않아요. 컨트롤러가 Reconcile 루프를 돌아야 생깁니다.

 # Ginkgo + Gomega
 ## 핵심 장점

### Eventually() 
비동기 폴링은 K8s operator 테스트에서 가장 중요한 기능입니다. 
``Eventually(func() { ... }, 타임아웃, 폴링간격)``은 지정된 시간 동안 반복해서 조건을 체크합니다. Reconcile이 완료될 때까지 기다린다. ``testing.T``로 이걸 하려면 ``time.Sleep``이나 직접 ``retry 루프``를 짜야 하는데, 불안정하고 코드가 지저분해집니다.

### Describe/Context/It 계층 구조
테스트가 212개나 되는 controller_test.go에서, **TLS 시나리오 → 인증서 재사용 → 콘솔과 acceptor가 같은 시크릿 공유**처럼 의도를 자연어에 가깝게 표현할 수 있습니다. ``testing.T``로 212개 테스트를 관리하면 ``TestTlsSecretReuseConsoleAndAcceptorShareOneSecret`` 같은 긴 함수명이 되고, 계층 관계가 사라집니다.

### BeforeEach/AfterEach 라이프사이클
각 It 블록 실행 전후로 CR 생성/정리를 자동화합니다. 이 프로젝트에서는 BeforeEach에서 네임스페이스와 기본 CR을 셋업하고, AfterEach에서 정리하는 패턴을 씁니다.

## 테스트 커버리지 측정 방법

### 환경
```
CRD가 15개 정도 있고
각 controller가 ./internal/{CRD}/{CRD}_controller.go
각 폴더에 suite_test.go, {CRD}_controller_test.go
CRD끼리 ownerRef, status, spec 반영 등으로 서로 영향을 줌
```

### 한 CRD 테스트는 그 controller의 책임을 중심으로 작성
예를 들어 resource_controller.go를 테스트한다면,

>ResourceReconciler가 입력 상태를 받았을 때, 자기 책임 범위 안에서 무엇을 보장해야 하는가?

즉 테스트의 중심은 **해당 controller의 책임**이다.
다른 CRD는 그 책임을 검증하기 위한 전제조건 / 협력 객체로 둔다.

예를 들면 Resource controller 테스트에서는 다음과 같이 역할을 분리한다.

> API가 있어야 Resource가 reconcile 가능하다면 → API는 fixture

> Method가 생성되어야 한다면 → 결과 검증 대상

> APIDeployment가 갱신되어야 한다면 → side effect 검증 대상

### 한 CRD 테스트 안에서 검증할 것
1. Entry 시나리오

: controller가 reconcile을 시작하게 되는 입력상태 

2. Core responsibility

: controller가 직접 해야하는 일

3. Side effects 
 
: 다른 CRD에 어떤 영향을 줘야하는지

4. Boundary / failure behavior

: 잘못된 입력이나 중간 실패에서 어떻게 독작해야 하는지

### 테스트 레벨 분리
1) Pure/unit test

- Kubernetes API 없이 순수 Go 로직만 테스트
- 헬퍼 함수, status 계산 함수, tree 구성 함수
- buildMethodList()
- computePhase()
- validateRefs()

2) Single-controller envtest
 
- 실제 Kubernetes API server + etcd를 띄우고 controller 하나만 테스트
- controller-runtime의 envtest 사용.
- reconcile 로직, finalizer 처리, ownerReference 설정, status Update, child resource 생성
```
CR 생성
--> 
특정 controller만 reconcile
--> 
해당 controller의 side effect 검증
```
예를 들어 ResourceController 테스트를 한다면

#### Given

> API CR 생성
>
> Resource CR 생성
>
> Method CR 생성

#### When

> Resource controller reconcile

#### Then

> Resource.status.methodList 업데이트

#### 코드 흐름
```
BeforeEach(func() {
    k8sClient.Create(ctx, api)
    k8sClient.Create(ctx, resource)
    k8sClient.Create(ctx, method)
})

It("updates resource status", func() {

    Eventually(func() string {
        updated := &Resource{}
        k8sClient.Get(ctx, resourceKey, updated)

```

3) Multi-controller integration test
- 여러 controller를 동시에 실행해서 실제 operator 흐름 테스트
- API → Resource → Method → APIDeployment 같이 실제 연결 흐름 검증

예시 흐름

#### Given

> API 생성

#### When

> API controller → Resource 생성
>
> Resource controller → Method 연결
>
> Method controller → status 업데이트
>
> Deployment controller → deployment 생성

Then

API.status가 전체 tree 상태를 반영

## suite_test.go
1) 테스트 진입점
```
func TestControllers(t *testing.T) {
	RegisterFailHandler(Fail)
	RunSpecs(t, "API Controller Suite")
}
```
2) 공통 테스트 환경 초기화
```
envtest.Environment 생성

CRD 로드

cfg 생성

k8sClient 생성

ctx, cancel 생성

test namespace 준비
```
이걸 모든 It(...)에서 매번 다시 하려면 너무 무겁고 중복되느로 BeforeSuite에서 한번에 진행

BeforeEach와의 차이 : BeforeSuite는 패키지 전체에서 한 번만 실행(envtest start, manager, start, CRD load 등 무거운 것 넣기 좋음)
BeforeEach는 각 It마다 실행(namespace 생성, fixture 객체 생성, mock 값 초기화)

3) 공통 정리
```
envtest stop

goroutine 종료

cancel 호출
```

하나의 폴더에 테스트 파일이 여러개 있어도 ``suite_test.go`` 는 보통 하나

여러 controller test에서 공통 helper 함수를 같이 쓰고 싶다면 새로운 파일을 만들어서 (test_helper.go 등) 관리하는 것이 좋음. --> suite_test.go는 환경 부팅이 주 목적

>resource가 생성되면 finalizer를 붙인다.

>

## suite_test.go 역할
> - Ginkgo 테스트 시작점
> - envtest 실행
> - scheme 등록
> - k8s client 생성
> - manager 생성
> - Resource controller 등록
> - 테스트 종료 cleanup

```
package resource_test

import (
	"context"
	"path/filepath"
	"runtime"
	"testing"
	"time"

	apiv1 "example.com/operator/api/v1alpha1"
	resourcecontroller "example.com/operator/internal/resource"

	. "github.com/onsi/ginkgo/v2"
	. "github.com/onsi/gomega"

	corev1 "k8s.io/api/core/v1"
	"k8s.io/apimachinery/pkg/runtime"
	utilruntime "k8s.io/apimachinery/pkg/util/runtime"
	clientgoscheme "k8s.io/client-go/kubernetes/scheme"

	ctrl "sigs.k8s.io/controller-runtime"
	"sigs.k8s.io/controller-runtime/pkg/client"
	"sigs.k8s.io/controller-runtime/pkg/envtest"
	"sigs.k8s.io/controller-runtime/pkg/log/zap"
)

// Ginkgo 테스트 시작점
func TestResourceController(t *testing.T) {
	RegisterFailHandler(Fail)
	RunSpecs(t, "Resource Controller Suite")
}

var (
	ctx       context.Context
	cancel    context.CancelFunc
	testEnv   *envtest.Environment
	k8sClient client.Client
	testScheme *runtime.Scheme
)

// 패키지 전체에서 한 번만 실행
// envtest start
// 
// scheme 등록
// 
// client 생성
//
// manager/controller start
var _ = BeforeSuite(func() {
	ctrl.SetLogger(zap.New(zap.WriteTo(GinkgoWriter), zap.UseDevMode(true)))

	ctx, cancel = context.WithCancel(context.Background())

	testScheme = runtime.NewScheme()
	utilruntime.Must(clientgoscheme.AddToScheme(testScheme))
	utilruntime.Must(corev1.AddToScheme(testScheme))
	utilruntime.Must(apiv1.AddToScheme(testScheme))

	_, filename, _, _ := runtime.Caller(0)
	projectRoot := filepath.Dir(filepath.Dir(filepath.Dir(filename)))

	testEnv = &envtest.Environment{
		CRDDirectoryPaths: []string{
			filepath.Join(projectRoot, "config", "crd", "bases"),
		},
		ErrorIfCRDPathMissing: true,
	}

	cfg, err := testEnv.Start()
	Expect(err).NotTo(HaveOccurred())
	Expect(cfg).NotTo(BeNil())

	k8sClient, err = client.New(cfg, client.Options{Scheme: testScheme})
	Expect(err).NotTo(HaveOccurred())
	Expect(k8sClient).NotTo(BeNil())

	mgr, err := ctrl.NewManager(cfg, ctrl.Options{
		Scheme: testScheme,
	})
	Expect(err).NotTo(HaveOccurred())

	// Resource controller만 등록
	err = (&resourcecontroller.ResourceReconciler{
		Client: mgr.GetClient(),
		Scheme: mgr.GetScheme(),
	}).SetupWithManager(mgr)
	Expect(err).NotTo(HaveOccurred())

	go func() {
		defer GinkgoRecover()
		err := mgr.Start(ctx)
		Expect(err).NotTo(HaveOccurred())
	}()

	// manager 시작 안정화 대기
	time.Sleep(500 * time.Millisecond)
})

// suite 종료 시 한 번만 실행
// context cancel
//
// envtest stop
var _ = AfterSuite(func() {
	cancel()

	err := testEnv.Stop()
	Expect(err).NotTo(HaveOccurred())
})
```
